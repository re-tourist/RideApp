## 生成《项目开发现状总览.md》

你是本项目的“技术文档工程师 + 架构梳理助手”。请基于我提供的代码仓库（包含前端 Android 客户端/后端 SpringBoot/文档等），生成一份**单一 Markdown 文档**：`PROJECT_STATUS.md`（或 `项目开发现状总览.md`）。

### 0. 目标与读者

- 目标：用一份文档让新成员/老师/评审在 10 分钟内理解**我们现在做到了什么、代码在哪里、怎么跑、关键实现方式、已知问题与可优化点**。
- 读者：项目组同学 + 助教/老师 + 后续接手的人 + 另一个 AI code agent。

### 1. 输出要求（强制）

- 只输出 **一个 Markdown 文件内容**（不要输出其它解释）。
- 结构清晰：必须包含目录（TOC）。
- 所有结论要“可追溯”：每个模块都要指向具体文件路径（相对路径）。
- 不要泛泛而谈：每个模块至少给出
  1. **做了什么**
  2. **关键类/函数/数据结构**
  3. **实现方法/关键流程**
  4. **当前状态：已完成/部分完成/未开始/假数据**
  5. **可优化点（性能/结构/可维护性）**

### 2. 你需要先做的扫描步骤（在内部完成，不要在输出里写过程）

1. 扫描仓库根目录文件树（深度建议 4-6 层），识别：
   - Android 客户端模块（Compose/传统 View）、网络层、数据层、页面模块
   - 后端服务模块（controller/service/repository/entity/dto/config）
   - 数据库脚本/ER 图/接口文档/PRD/MRD 等 docs
2. 识别“核心业务域”并分组：比如用户、骑行记录、路线、社交动态、活动/赛事、俱乐部、商城/二手、消息通知等。
3. 找出“入口与运行路径”：Android 启动页/导航图；后端启动类与端口、配置文件；数据库连接方式。
4. 找出“网络调用链路”：API 定义 → DTO/BTO → Repository/ApiKit → ViewModel → UI。

### 3. 文档固定大纲（必须按此顺序输出）

#### A. 项目一句话介绍 & 当前里程碑

- 一句话介绍（产品定位 + 技术栈）
- 里程碑表（建议用表格）：模块 / 目标 / 当前进度 / 负责人（如能从提交者推断）/ 风险

#### B. 快速开始

- 环境依赖（JDK、Android Studio、Gradle、MySQL 等）
- 如何启动后端（命令、配置文件位置、端口、健康检查方式）
- 如何启动 Android（运行配置、BASE_URL 配置位置、必要权限）
- 启动顺序（如有）
- 常见启动失败排查（列 5-10 条最常见的）

#### C. 仓库文件结构总览（非常重要）

- 用树形结构展示（只要关键目录）
- 每个一级目录一句话说明用途
- 标出“最常改动的目录”“危险目录（改动易出问题）”“可清理的目录（如发现无用）”

#### D. 架构与数据流

- 客户端架构：UI(Compose) / ViewModel / Repository / Network / LocalCache（按实际项目写）
- 后端架构：Controller / Service / Repository / Entity / DTO / Security / Config
- 端到端数据流示例（至少 2 条）
  - 示例 1：登录/注册链路
  - 示例 2：骑行记录列表/详情链路（或你仓库里最完整的一条链路）

#### E. 模块级现状盘点（逐模块写）

对每个模块都用同一模板输出：

**模块：XXX（例如：用户与登录）**

- 功能范围：……
- 关键页面/接口：……
- 关键文件清单（必须列路径 + 一句话）：
  - `path/to/FileA`：做什么
  - `path/to/FileB`：做什么
- 关键实现方法（面向“怎么做的”）：
  - 例如：token 存储方式、拦截器、错误码处理、分页方式、状态管理方式等
- 当前完成度：已完成/部分完成/未完成（说明缺口）
- 已知问题/风险：
  - 例如：假数据、接口字段不一致、性能瓶颈、重复代码、潜在崩溃点
- 优化建议（必须可执行，给出“改哪里”）：
  - 结构优化：例如抽公共组件/封装 ApiResult/统一 error handling
  - 性能优化：例如冷启动、延迟加载、图片缓存、列表分页、数据库索引、N+1 查询
  - 工程化：例如 lint、ktlint、detekt、CI、commit 规范、PR 模板

模块建议至少覆盖（按你仓库实际有的来，没做也要写“未开始”）：

- 用户/登录注册/个人资料
- 骑行记录（开始/暂停/结束/轨迹/GPS）
- 路线/路书（创建/收藏/导航）
- 社区动态（发帖/点赞/评论/关注）
- 活动/赛事（列表/详情/报名）
- 俱乐部（创建/加入/成员管理）
- 商城/二手交易（如果只是链接也要写清楚）
- 消息/通知
- 设置/隐私/权限

#### F. 接口与数据模型对齐情况

- 列出后端已实现的 API 清单（路径、方法、用途、是否联调通过）
- 列出客户端已接入的 API 清单（对应到 UI 页面）
- 字段一致性检查：找出明显不一致的 DTO 字段（能列多少列多少）

#### G. 性能与体验现状（必须写“现状 + 证据 + 优化路线”）

- 已观察到的卡顿/加载慢点（例如：打开页面要等几秒才出数据）
- 可能原因（只基于代码推断，别编）
- 优化路线（按优先级 P0/P1/P2）：
  - P0：立刻见效（缓存、骨架屏、并发、分页、减少主线程工作）
  - P1：需要结构调整（数据层改造、统一状态流）
  - P2：长期工程化（监控、性能基准、CI）

#### H. 测试与联调现状

- 当前有哪些测试（单元/UI/接口），没有也要明确写“暂无”
- 联调方式：Postman/Swagger/日志
- 建议的最小可行测试集（列 10 条用例）

#### I. 下一步计划（可直接拆任务）

- 用表格：任务 / 目标 / 涉及文件 / 预估难度（S/M/L）/ 风险
- 优先列出 10-20 条“最该做”的

### 4. 写作风格要求

- 用“项目内能直接执行”的表达：多写路径、类名、函数名、字段名。
- 避免空话（如“提高性能、优化体验”必须落到改动点）。
- 对不确定的地方必须标注：`(推测)` 或 `TODO: 需确认`，不要编造。

### 5. 交付

- 最终只输出 Markdown 正文内容。
- 文件名写在文档开头第一行标题里，例如：`# 项目开发现状总览（PROJECT_STATUS.md）`
